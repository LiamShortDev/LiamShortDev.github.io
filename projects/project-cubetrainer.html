Rubik's Cube Solver & Trainer

Tech Stack

  • C#
  • WinForms
  • 2D Projection for Cube Visualization
  • SQL Databases
  • Hashing Algorithms (SHA-256)
  • Heuristic, Recursive Shortest-Path Algorithm

Why I Built This

I’ve been fascinated by Rubik’s cubes for a while — not just solving them quickly, but understanding the logic behind speed-solving methods. So when it came to my A-Level Computer Science project, I saw the perfect opportunity to combine that passion with programming. I approached a friend who’d mentioned wanting to learn how to solve the cube faster, and offered to build a tool that could help. After chatting with them about their struggles and goals, I put together a clear set of targets: the software needed to track solve times, suggest algorithms, and provide meaningful feedback to help them improve.

Detailed Analysis

The application was built using WinForms for the user interface, with an SQL database for storing solve times, CFOP algorithms (both standard and user-defined), and securely hashed login credentials using SHA-256.

The core of the trainer is its interactive solver, which accepts a scrambled cube and solves it using the human CFOP method. The solve is broken into four stages, allowing the user to step through and understand each part of the process. To handle the white cross stage efficiently, I implemented a heuristic shortest-path algorithm inspired by A*, which consistently finds near-optimal solutions in a fraction of a second.

Users can browse any standard CFOP algorithm or their own added algorithms, with step-by-step visualisation. This helps users both learn when to use each algorithm and become familiar with the Rubik’s Cube notation.

I chose a 2D projection to display the cube, calculated using basic trigonometry. While a 3D representation with mouse interaction would have been more intuitive, the 2D model kept the project within scope and intentionally reinforced learning notation — a key skill for more advanced solving methods.

Challenges Faced

One of the biggest challenges was building a solver that produced realistic solutions using the CFOP method — not just solving the cube, but solving it in a human-understandable way. Initially, the solver relied on brute force and generated solutions that were far too long to be practical. Adapting a traditional A* algorithm to suit this use case required a lot of experimentation with heuristics to prioritize move efficiency over speed of execution.

Another challenge was designing a Rubik’s Cube UI in a 2D environment. Figuring out how to accurately represent the cube’s state visually — and make it intuitive — took time and trial and error, especially when mapping face rotations to button inputs.

Lessons Learned

This project taught me how to balance functionality with usability — not just building something that works, but building something people will actually enjoy using. I gained experience with algorithm optimization, user interface design in WinForms, and working with SQL databases to persist user data securely.

I also learned the importance of scoping a project realistically. My initial ideas were too ambitious, and I had to prioritise the features that delivered the most learning value and practical benefit within the available timeframe.

If I Did It Again...

With more time and experience, I’d implement a full 3D interactive cube using a graphics library to allow users to manipulate the cube naturally with drag-and-rotate controls. I’d also consider building the app cross-platform, possibly using a framework like WPF, Qt, or even converting it into a web-based tool so it’s more accessible.

I’d also improve the algorithm viewer to highlight move patterns visually and help users connect algorithms to actual cube states more intuitively — bridging the gap between memorization and recognition.
